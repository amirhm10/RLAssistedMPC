# Distillation Warm-Start Training Analysis

Date: 2026-04-23

This note studies the current distillation `matrix` and `residual` TD3 workflows from three angles:

1. What the code is actually doing during warm start.
2. Why that behavior is already a narrow offline-to-online pretraining scheme.
3. What the RL literature says about using such a scheme to reduce degradation, and what the next implementation step should be.

The supporting repo figures were generated by `report/scripts/distillation_warm_start_analysis.py`.

As part of this refresh, the TD3 notebook defaults were also updated so `policy_delay = 2` across the shared polymer and distillation notebook parameter tables, and the standalone `RL_assisted_MPC_Poles.ipynb` TD3 setup was aligned to the same default.

## Executive Summary

- The current distillation `matrix` and `residual` runners already use warm start as replay-buffer prefill from baseline MPC behavior.
- The learned policy is held at the nominal warm-start action until `i > warm_start_step`, but warm-start transitions are still inserted into replay and `train_step()` starts at `i >= warm_start_step`.
- Therefore, the first actor/critic update is computed on warm-start data before the first learned action is actually deployed.
- That means the repo already performs a genuine offline-like actor-critic bootstrap before online release.
- This is a valid idea. The literature strongly supports replay seeding, offline-to-online fine-tuning, and actor regularization toward dataset behavior.
- The literature does not support plain unconstrained actor maximization on a narrow warm-start dataset as the best way to avoid degradation.
- The most defensible next step is not "train more during warm start," because the repo already does that. The next step is "make the warm-start-to-online handoff more conservative."

## Mathematical View Of The Current Scheme

Define the warm-start boundary as `tau = warm_start_step`.

Let the warm-start behavior policy be:

$$
\beta_{\text{ws}}(a \mid s) =
\begin{cases}
\delta(a - a_0(s)), & t \le \tau \\
\text{learned policy rollout}, & t > \tau
\end{cases}
$$

where:

- for `matrix`, $a_0(s) = 0$ in normalized action coordinates, which maps to nominal multipliers
- for `residual`, $a_0(s) = a_{\text{zero-residual}}$, which maps to zero additive correction

The warm-start replay dataset is therefore:

$$
\mathcal{D}_{\text{ws}}=\{(s_t, a_0(s_t), r_t, s_{t+1})\}_{t=0}^{\tau}.
$$

At the release boundary, the current implementation performs TD3 training with:

$$
\mathcal{D}_{4000}\subseteq \mathcal{D}_{\text{ws}}.
$$

So the first actor/critic update is fully offline with respect to the current policy release. That is the key reason the current code is already pretraining-like.

### Current critic objective

For the non-sequence TD3 path in this repo, the critic target is:

$$
y_t = r_t + \gamma^n Q_{\phi^-}\!\left(s_{t+1},
\pi_{\theta^-}(s_{t+1}) + \epsilon\right),
\qquad
\epsilon \sim \text{clip}\left(\mathcal{N}(0,\sigma_{\text{targ}}^2), -c, c\right),
$$

and the twin critics minimize:

$$
\mathcal{L}_{\text{critic}}(\phi)
=
\mathbb{E}_{(s,a,r,s')\sim \mathcal{D}}
\left[(Q_{\phi_1}(s,a)-y)^2 + (Q_{\phi_2}(s,a)-y)^2\right].
$$

### Current actor objective

The TD3 actor update is:

$$
\mathcal{L}_{\text{actor}}(\theta)
=
-\mathbb{E}_{s\sim \mathcal{D}}[Q_{\phi_1}(s,\pi_\theta(s))].
$$

This is where the warm-start issue appears.

During the first actor updates, the states $s$ come from baseline MPC trajectories, but the actor is free to choose actions not represented by the narrow warm-start behavior policy. That creates a classic support mismatch:

$$
\pi_\theta(s)\not\approx a_0(s)
\quad\text{while}\quad
Q_\phi(s,a)\text{ has mostly been trained near }a_0(s).
$$

That is the precise mathematical reason why warm-start replay alone may not prevent degradation.

## Why Degradation Can Still Happen

At the release boundary, the first online actor update is still based on the warm-start dataset:

$$
\theta_{1}
\leftarrow
\arg\max_\theta
\mathbb{E}_{s\sim \mathcal{D}_{\text{ws}}}
Q_\phi(s,\pi_\theta(s)).
$$

But the actor is not asked to remain close to $a_0(s)$. If the critic extrapolates too optimistically away from the warm-start support, the actor can move into directions that are:

- not well covered in replay
- not yet validated on the real rollout
- larger than the safe warm-start behavior

This is exactly the offline RL failure mode highlighted by conservative and behavior-regularized methods:

- overestimation outside the data support
- actor drift toward critic hallucinations
- post-release degradation before replay catches up

In this repo, that logic is especially relevant because warm-start data are intentionally narrow:

- matrix warm start stays at the nominal model
- residual warm start stays at zero correction

That is useful for safety and stability, but it also means the first actor updates are trained on a thin support around the baseline MPC manifold.

## Literature Support

The idea of using warm-start data to bootstrap actor-critic learning is valid and well supported.

### Primary papers used here

- Vecerik et al., 2017: replay seeded with demonstrations for DDPG-like learning
- Nair et al., 2018: demonstrations used to solve early exploration bottlenecks
- Nair et al., 2021 (AWAC): offline pretraining plus online fine-tuning
- Fujimoto et al., 2018 (TD3): delayed actor updates and overestimation control
- Kumar et al., 2020 (CQL): conservative critics for offline data
- Fujimoto and Gu, 2021 (TD3+BC): actor regularization toward dataset actions
- Cetin et al., 2024: simple offline RL methods remain effective when scale and downstream-task alignment are handled well

### What the papers say, distilled

#### 1. Replay seeding and demonstrations are useful

Vecerik et al. write that demonstrations and actual interactions are both used to fill replay, with prioritized sampling, and that this reduces the exploration problem in continuous-control robotics tasks. Nair et al. similarly report order-of-magnitude speedups when demonstrations are used to overcome exploration bottlenecks.

This supports the first half of the repo’s current design:

- baseline MPC warm-start data are useful
- warm-start replay prefill is not just reasonable, it is directly aligned with prior practice

#### 2. Offline-to-online fine-tuning is a valid paradigm

AWAC explicitly studies pretraining from offline datasets followed by online fine-tuning. Its central point is that this is useful, but naive transfer is difficult.

This supports your intuition:

- yes, warm-start training is a valid offline-to-online idea
- no, it is not enough to just train during warm start and hope the actor release is stable

#### 3. Plain actor-critic can drift outside the data support

TD3 identifies actor-critic overestimation and motivates delayed actor updates. CQL shows that standard off-policy RL can fail on fixed datasets due to overestimation induced by distribution shift between the dataset and the learned policy.

This is directly relevant to the current code because the warm-start data support is narrow.

#### 4. Actor regularization is a strong next step

TD3+BC shows that a plain TD3 actor can be stabilized in the offline regime by adding a behavior-cloning term to the actor objective. Cetin et al. show that simple methods such as AWAC and IQL can remain strong when the dataset is aligned with the downstream task and scale is handled properly.

This strongly suggests that if the repo wants to exploit warm-start data better, the missing ingredient is actor regularization during the handoff, not just more replay.

## Mathematical Form Of The Most Relevant Alternatives

### A. Critic-only release

Keep the current replay prefill but freeze the actor for `K` post-warm-start episodes:

$$
\theta_{k+1} = \theta_k
\quad\text{for}\quad
\tau < t \le \tau + K H,
$$

while the critic still updates on replay:

$$
\phi_{k+1}
\leftarrow
\arg\min_\phi
\mathcal{L}_{\text{critic}}(\phi).
$$

This uses warm-start and early online data to stabilize the critic before the actor starts exploiting it.

### B. TD3+BC-style warm-start actor

Replace the plain actor objective during a handoff window by:

$$
\mathcal{L}_{\text{actor}}^{\text{BC}}(\theta)
=
-\mathbb{E}_{s\sim \mathcal{D}}[Q_{\phi_1}(s,\pi_\theta(s))]
+
\lambda_{\text{BC}}
\mathbb{E}_{(s,a)\sim \mathcal{D}_{\text{ws}}}
\left[\|\pi_\theta(s)-a\|_2^2\right].
$$

For this repo:

- matrix: $a$ is the nominal multiplier action
- residual: $a$ is the zero residual action

This is the simplest mathematically clean fix for the current support mismatch.

### C. AWAC-style handoff

Use advantage-weighted imitation rather than raw actor maximization:

$$
\max_\theta
\mathbb{E}_{(s,a)\sim \mathcal{D}}
\left[
\log \pi_\theta(a\mid s)
\exp\!\left(\frac{A^\pi(s,a)}{\lambda}\right)
\right].
$$

This keeps the actor close to actions that are already supported by data while still favoring high-advantage behavior.

### D. CQL-style critic

Use a conservative Q-function during warm start:

$$
\mathcal{L}_{\text{CQL}}
=
\mathcal{L}_{\text{Bellman}}
+
\alpha
\left(
\mathbb{E}_{s,a\sim \pi_\theta}[Q_\phi(s,a)]
-
\mathbb{E}_{s,a\sim \mathcal{D}}[Q_\phi(s,a)]
\right).
$$

This penalizes optimistic Q-values on actions not well represented in the warm-start dataset.

### E. Authority or range ramp

For residual:

$$
u_t = u_t^{\text{MPC}} + \rho_t \Delta u_t^{\text{RL}},
\qquad
\rho_t \uparrow 1.
$$

For matrix:

$$
a_t = (1-\rho_t)a_0 + \rho_t a_t^{\text{RL}},
\qquad
\rho_t \uparrow 1,
$$

or equivalently ramp the admissible multiplier bounds away from nominal after warm start.

This does not solve critic extrapolation directly, but it reduces the damage if the first released actor steps are poor.

## What The Next Step Should Be

The next step should be a conservative warm-start release, not a larger warm-start window.

The repo already has:

- safe warm-start data generation
- replay prefill
- pre-release actor/critic training

So the most useful next step is to make the actor handoff conservative.

## Evaluating The Proposed Phase 1 Variant

The proposal to think through is:

1. keep replay filling during warm start
2. possibly let the critic learn earlier
3. after warm start, keep the executed action frozen for about 5 sub-episodes
4. train in the background
5. then release full TD3 control

This needs to be split into two separate design questions, because they are not equally sound.

### Clarification: the current code already disables learning during replay filling

In both active distillation runners, replay filling during the warm-start window is already pure data collection:

- `utils/matrix_runner.py:375`
- `utils/residual_runner.py:393`

Both runners only call `agent.train_step()` when:

$$
i \ge \tau = \texttt{warm\_start\_step}.
$$

So for the current code:

- replay filling during steps `0..3999` already has no critic learning
- replay filling during steps `0..3999` already has no actor learning

That means the question is not whether to disable critic learning during warm-start replay fill. That is already the current behavior.

### Question A: should critic learning start during warm-start replay filling?

In principle, critic-only warm-start pretraining is a valid idea. It would move the design closer to explicit offline pretraining on baseline MPC data.

However, with the current TD3 implementation, the critic target is still actor-dependent:

$$
y_t = r_t + \gamma^n Q_{\phi^-}\!\left(s_{t+1}, \pi_{\theta^-}(s_{t+1}) + \epsilon\right).
$$

So if we start critic learning early while the actor and target actor are still essentially untrained, the critic is not doing pure evaluation of the baseline warm-start policy. It is bootstrapping through:

- warm-start data actions near $a_0(s)$
- but target actions near $\pi_{\theta^-}(s)$, where $\pi_{\theta^-}$ is not yet aligned with $a_0$

That creates the mismatch:

$$
(s_t, a_t) \sim \mathcal{D}_{\text{ws}}
\quad\text{with}\quad
a_t \approx a_0(s_t),
\qquad
y_t \text{ bootstraps through } \pi_{\theta^-}(s_{t+1}).
$$

So enabling critic learning during replay fill is not obviously wrong, but it is not the cleanest first change in this repo. It becomes much more defensible only if we also anchor the target action during that phase, for example by using:

$$
y_t^{\text{base}}
=
r_t + \gamma^n Q_{\phi^-}\!\left(s_{t+1}, a_0(s_{t+1})\right),
$$

or by explicitly initializing / constraining the actor toward the warm-start action.

Conclusion for Question A:

- not my first Phase 1 change with the current TD3 code
- reasonable later, if paired with a baseline-anchored target action or BC-style actor anchoring

### Question B: should we freeze the executed action for 5 sub-episodes after warm start, but let actor training run?

This is the more important part of your proposal, and here the answer is:

- freezing the executed action is good
- letting actor optimization continue behind the scenes during that hidden window is not the best Phase 1

Why?

During a hidden-action window of length $K$ sub-episodes, the data-generating behavior is still baseline MPC:

$$
\beta_{\text{hidden}}(a \mid s) = \delta(a-a_0(s)),
\qquad
\tau < t \le \tau + K H,
$$

so the accumulated dataset remains concentrated near $a_0(s)$:

$$
\mathcal{D}_{\text{hidden}}
\approx
\mathcal{D}_{\text{ws}} \cup \mathcal{D}_{a_0}.
$$

But if actor training is enabled in the background, TD3 still does:

$$
\theta_{k+1}
\leftarrow
\arg\max_\theta
\mathbb{E}_{s\sim \mathcal{D}_{\text{hidden}}}
Q_\phi(s,\pi_\theta(s)).
$$

The key problem is that the actor is free to move away from $a_0(s)$ even though:

- the replay still contains almost only baseline-action data
- the plant has still not executed those learned actions
- the critic has still not seen the consequences of those learned actions

So the quantity

$$
d_{\text{release}}(\theta)
=
\mathbb{E}_{s\sim \mathcal{D}_{\text{hidden}}}
\left[\|\pi_\theta(s)-a_0(s)\|_2\right]
$$

can grow during the hidden window. If that happens, the first released action after 5 sub-episodes may jump further away from baseline than it does now.

That is why "freeze action, but still train the actor" is not the safest interpretation of Phase 1. It can delay the deployment shock in time, but it can also make the eventual release jump larger.

### What I recommend instead

For this repo, the better Phase 1 is:

1. keep the current warm-start replay fill as-is
2. after warm start, keep the executed action frozen at the warm-start action for 5 sub-episodes
3. keep collecting those baseline-style transitions into replay
4. keep critic training active during those 5 sub-episodes
5. keep actor optimization frozen during those 5 sub-episodes
6. only then release actor updates and learned actions

Mathematically, the recommended hidden window is:

$$
\phi_{k+1}
\leftarrow
\arg\min_\phi \mathcal{L}_{\text{critic}}(\phi),
\qquad
\theta_{k+1}=\theta_k,
\qquad
u_t = u_t^{\text{warm-start}}
\quad\text{for}\quad
\tau < t \le \tau + 5H.
$$

This is much closer to a true "critic stabilization before release" phase.

### Important implementation note for later

The existing TD3 `actor_freeze` knob in `TD3Agent/agent.py` is useful, but it is not the whole Phase 1 mechanism:

- it is measured in training steps, not in sub-episodes
- for the current distillation setup, 5 sub-episodes means about `5 * 400 = 2000` train steps
- it freezes the actor optimizer step, but action-freeze still has to be implemented in the runner

So if we implement this later, we should treat it as two coordinated controls:

- runner-level executed-action freeze
- agent-level actor optimizer freeze

### Bottom line on the proposal

If your intended idea is:

- "during warm-start replay fill, still no learning"
- "after warm start, freeze the executed action for 5 sub-episodes"
- "train critic only in the background"
- "then release full TD3"

then yes, that is a good Phase 1 and it is the one I would recommend.

If your intended idea is instead:

- "after warm start, freeze the executed action for 5 sub-episodes"
- "but let actor optimization continue during those 5 sub-episodes"

then I do not recommend it as the first change. It is more likely to store up actor drift than to remove the release problem.

## Recommended Next Implementation

The best first implementation for this repo is:

### Phase 0. Keep the current warm-start replay prefill

No change:

- matrix uses nominal multipliers during warm start
- residual uses zero residual correction during warm start
- all warm-start transitions are pushed into replay

### Phase 1. Critic-only hidden release window

Add a config:

- `post_warm_start_action_freeze_subepisodes`
- `post_warm_start_actor_freeze_subepisodes`

Behavior:

- after warm start ends, keep the executed action frozen at the warm-start action for `K` sub-episodes
- set `K = 5` for the first distillation Phase 1 trial
- keep actor updates frozen during that same hidden window
- critic still trains normally on the newly accumulated baseline-style replay
- after that window, release the actor and optionally use a conservative authority ramp

### Phase 2. Add TD3+BC anchor during the handoff window

Add configs:

- `warm_start_bc_weight`
- `warm_start_bc_decay_subepisodes`
- `warm_start_bc_dataset = "warm_start" | "mixed_recent"`

Use:

$$
\mathcal{L}_{\text{actor}}
=
-Q
+
\lambda_{\text{BC}} \|\pi(s)-a_{\text{anchor}}(s)\|^2
$$

with:

- matrix anchor = nominal multiplier action
- residual anchor = zero residual action

### Phase 3. Release authority gradually

Add configs:

- `post_warm_start_authority_ramp_subepisodes`
- `post_warm_start_authority_rho_floor`

For residual:

- ramp residual authority from a smaller $\rho$ to full authority

For matrix:

- ramp effective multiplier range from narrower to full configured range

This is a strong fit for the distillation case because the early degradation is a release problem, not a warm-start-data problem.

## Final Conclusion

Warm-start training is a valid idea.

The current distillation matrix and residual implementations already do it.

But the real bottleneck is now the handoff:

- the actor is first trained on narrow baseline-MPC support
- then it is released online with too little constraint

So the next meaningful step is:

- keep warm-start replay prefill
- keep critic training
- make actor release conservative

The best first implementation is:

- critic-only post-warm-start freeze
- TD3+BC anchor during handoff
- optional authority/range ramp

That is more technically justified than simply enlarging warm start or further increasing plain TD3 training on the same narrow warm-start dataset.
