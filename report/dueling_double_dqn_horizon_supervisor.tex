\section{Dueling Double DQN Horizon Supervisor}

\subsection{Motivation}
The horizon-selection problem in RL-assisted MPC is a discrete supervisory control task: the agent selects a horizon recipe $(H_p, H_c)$ while the underlying offset-free MPC handles the continuous control move optimization. Near steady state, many horizon recipes can yield similar closed-loop performance, which makes a value--advantage decomposition attractive. In that setting, the state can still have high value even when the relative advantage between candidate horizon recipes is small. A dueling architecture is therefore a natural extension of the current discrete horizon supervisor.

\subsection{Method}
Let the discrete action $a$ denote a horizon recipe index. The recipe library maps each index to one pair $(H_p, H_c)$, and the MPC solve uses that pair for the current supervisory interval.

For state $s$, the dueling network estimates
\[
V(s) \in \mathbb{R}, \qquad A(s,a) \in \mathbb{R}^{|\mathcal{A}|}.
\]
The action-value estimate is formed as
\[
Q(s,a) = V(s) + A(s,a) - \frac{1}{|\mathcal{A}|}\sum_{a'} A(s,a').
\]
This avoids the non-identifiability of the naive $Q = V + A$ form.

The Double DQN target is
\[
y = r + \gamma Q_{\text{target}}\!\left(s', \arg\max_a Q_{\text{online}}(s', a)\right)(1-d),
\]
where $d$ is the done flag. The online network selects the next action and the target network evaluates it. Training uses the Huber loss
\[
\mathcal{L} = \operatorname{SmoothL1}\!\left(Q_{\text{online}}(s,a), y\right),
\]
with importance weights from the replay buffer when prioritized replay is active.

\subsection{Mapping To This Repo}
The dueling workflow is implemented for the polymer horizon notebook. The state is the same unified horizon state already used in the repo:
\begin{itemize}
  \item standard mode: scaled observer state, scaled setpoint deviation, and scaled input deviation
  \item mismatch mode: the standard state plus innovation and tracking-error terms
\end{itemize}

The action is still a discrete horizon recipe index. The shared runner translates that index into $(H_p, H_c)$, rebuilds the MPC solver only when the selected horizon changes, and keeps the same reward path, decision interval logic, disturbance handling, and warm-start switch as the current unified horizon workflow.

The notebook entrypoint is \texttt{RL\_assisted\_MPC\_horizons\_dueling\_unified.ipynb}. It uses the new runner \texttt{run\_dueling\_dqn\_mpc\_horizon\_supervisor} and stores the same normalized bundle fields as the current horizon path, with optional dueling-specific diagnostics added when available.

\subsection{Comparison Against The Legacy DQN Path}
The legacy repo path under \texttt{DQN/} remains unchanged and continues to serve as the baseline. That legacy path uses a twin discrete-Q structure, but in practice the current horizon notebook configuration relies on \texttt{q1} for action selection and target combination, so the second branch is not used as a fully symmetric twin learner. The new implementation therefore uses a standard and internally coherent dueling Double DQN design instead of trying to preserve that legacy twin-head structure.

\subsection{Expected Benefits And Limitations}
\textbf{Potential benefits:}
\begin{itemize}
  \item cleaner representation when many horizon actions are nearly equivalent
  \item more interpretable diagnostics via separate value and advantage statistics
  \item standard Double DQN target construction for stable discrete control
\end{itemize}

\textbf{Limitations:}
\begin{itemize}
  \item the action space remains a finite recipe library, so performance still depends on the chosen horizon grid
  \item reward design remains a dominant driver of the learned policy
  \item MPC solve cost still dominates runtime even if the RL estimator improves
  \item dueling structure is helpful only if state value and action advantage are meaningfully separable in this task
\end{itemize}

\subsection{Hyperparameter Notes}
The new defaults are chosen for slow supervisory MPC decisions rather than high-frequency benchmark environments:
\begin{itemize}
  \item linear epsilon decay for predictable exploration across a limited number of supervisory decisions
  \item hard target updates by default for stable discrete Q-learning targets
  \item Huber loss and gradient clipping for robustness to occasional large TD errors
  \item reuse of the existing prioritized-plus-recent replay buffer to stay compatible with the current repo's experience distribution design
\end{itemize}
